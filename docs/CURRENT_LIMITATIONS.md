# CURRENT_LIMITATIONS

## Summary

`ProtoGemCouch` has reached a successful demo milestone, but it is not yet fully wire-compatible with a native Apache Geode server.

The current implementation supports an effective demo path for string-like values and backend CRUD behavior against Couchbase, but several protocol and type-fidelity gaps remain.

---

## 1. Destroy / Remove Reply Wire Compatibility

### Current state
The shim successfully processes remove requests and deletes documents from Couchbase.

However, the Apache Geode client still does not fully accept the destroy/remove reply generated by the shim.

### Observed behavior
- `RemoveHandler` executes successfully
- Couchbase delete succeeds
- the client throws a `ServerConnectivityException` / destroy response parsing failure
- the sample demo works around this by catching the exception, reconnecting, and verifying the document is gone

### Impact
- backend delete behavior is correct
- native Geode protocol compatibility for destroy/remove is incomplete

### Next phase
Implement full native Geode wire compatibility for destroy/remove replies.

---

## 2. Response Type Fidelity Is Incomplete

### Current state
Some responses that a Geode client would normally expose as native Java types are currently returned in a simplified or demo-safe form.

Examples:
- string values may be returned to the client as `byte[]`
- boolean-style replies such as contains-key are not yet fully native in all cases

### Impact
- backend data is correct
- the sample app can still function with local decoding/workarounds
- client type fidelity is not yet production-grade

---

## 3. Geode DataSerializer Initialization Is Not Reliable in the Shim Runtime

### Current state
`org.apache.geode.DataSerializer` initialization fails in the shim runtime.

### Observed behavior
- deserialization attempts in request handling can throw initialization errors
- the project currently relies on fallback decoding for string-like payloads

### Impact
- current demo path works for string-like values
- general object serialization/deserialization compatibility is incomplete
- arbitrary complex Geode object types are not yet supported

---

## 4. Demo Path Uses Targeted Workarounds

### Current state
The sample app intentionally works around known protocol gaps to prove backend CRUD behavior.

Workarounds currently include:
- decoding returned `byte[]` values into strings
- using GET-based existence checks instead of strict Geode-native contains semantics
- catching the known delete response gap and reconnecting before verifying absence

### Impact
- this is acceptable for a milestone demo
- it is not the final client behavior target

---

## 5. Supported Data Shape Is Narrow

### Current state
The current validated path is primarily for:

- region + string key
- string-like value payloads

### Not yet validated
- arbitrary object graphs
- custom serialized types
- PDX objects
- collections / maps with full Geode serialization fidelity
- event metadata fidelity
- callbacks and listener semantics
- versioning / concurrency semantics beyond the current subset

---

## 6. Production Readiness Is Partial

### Already present
The project already includes major production-oriented building blocks such as:
- structured logging
- metrics
- startup validation
- benchmark and soak testing work
- deployment packaging
- runbooks and launch criteria

### Still missing for true production readiness
- full Geode wire compatibility for all supported operations
- broader compatibility testing with real Geode clients
- clearer supported/unsupported operation matrix
- stronger integration and regression coverage around real protocol captures
- final hardening around reconnect behavior, failures, and compatibility guarantees

---

## 7. Current Positioning

The project should currently be described as:

**A working Geode-to-Couchbase protocol shim prototype with successful end-to-end CRUD demonstration for string-like values, but not yet a fully native Geode-compatible server replacement.**

---

## Immediate Next Priority

The next highest-priority engineering task is:

## Full native Geode wire compatibility for destroy/remove replies

That work should include:
- capturing and comparing real Geode destroy/remove responses
- matching reply header and part structure exactly
- validating the destroy flow against a real Geode client without requiring any sample-app workaround
- promoting delete/remove from “backend-correct with client workaround” to “fully native client-compatible”

---

## Definition of Done for This Limitation Area

Destroy/remove compatibility will be considered complete when:

- `region.remove(...)` returns normally from a real Geode client
- no client-side protocol-gap exception is thrown
- the deleted document is absent in Couchbase
- no reconnect workaround is required
- the sample app can complete delete verification without special-case handling
